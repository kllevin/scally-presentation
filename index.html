How it came about?

Scally was created as an open source CSS framework for my old employer Westfield Labs.

It extended and nicely packaged up the CSS framework that I'd been developing and using over the many years working as a UI developer.

The name…

  • I was the lead front-end developer at Westfield Labs.
  • I introduced my own CSS framework to build UI mainly the new AU responsive website.
  • A year had gone by but the framework hadn't been updated so it was starting to feel neglected and lacked proper documentation.
  • I proposed that we needed to develop a proper CSS framework, WF agreed. I worked on it full-time for around 3 months collaborating with the rest of the team along the way and doing a shit load research.
  • Why roll your own? Inuit and SUIT CSS were the only 2 frameworks that came close to what I believed to be true OOCSS frameworks. Inuit was no longer being maintained and was showing it, and SUIT CSS used certain things in a way that I didn't quite agree with + it didn't use Sass.
  • I decided to name it Scally, which was a suggestion by my former head of engineering and friend Ian Tinsley, who like me is an Englishman, which is why it got named Scally.

What is it?

Bits and pieces to construct UI.

  • Building blocks so Lego™ is the best analogy for Scally and I will show some examples later in the presentation.

Scally is a collection of tiny, composable, configurable, interdependent modules that you can piece together how you want to, when you want to, giving you a solid, scalable CSS architecture.

  • All HTML elements render as blocks and pretty much all UI layout can be handled by a bunch of CSS rules abstracted away into reusable modules, this is what Scally does.

Key features

  • Highly modular following OOCSS principles.
  • Light weight: 37kb minified, 14kb minified and gzipped.
  • Responsive ready.
  • Unopinionated about design.
  • Uses Sass and BEM.
    • In a nutshell OOCSS basically means a CSS “object” is a repeating visual pattern, that can be abstracted into an independent snippet of HTML, CSS, and possibly JavaScript. That object can then be reused throughout a site. This allows us to build large-scale UI's (or any UI's for that matter) in a fast, maintainable, and standards-based way.
    • Two Main Principles of OOCSS:
      • Separate structure and skin
        This means not mixing in cosmetics (like background and border styles) but instead only including the structural CSS, this ties into point #4.
      • Separate container and content
        Essentially, this means “rarely use location-dependent styles”. An object should look the same no matter where you put it. Descendant selectors are a rare thing when building UI this way.
    • Scally applies all of this, it's there when you need it and getting out of your way when you don't.
    • I will show some examples later in the presentation.
  • Entire framework is 14kb gzipped however Scally allows you to only use what you need with the exception of the Core section therefore it can be as light as 6-8kb. This is a drop in the ocean compared to other site assets e.g. 1 jpeg can weigh anywhere between 20-100kb.
  • Is responsive ready making building responsive UI's easy. They're no absolute units used in Scally e.g. px, everything is fluid. Scally lets you change UI at breakpoints e.g. a grid can be 6 columns at super wide viewports, 3 columns at medium width viewports, and zero columns and small width viewports like handheld devices. However Scally doesn't force you to go responsive, you can easily create fixed width UI's with Scally.
  • Scally doesn't come with any design and everything is configurable. This approach means Scally acts as a solid foundation to your UI allowing you to layer on the cosmetic styles of the UI on top.
  • BEM is a front-end naming methodology. It is a smart way of naming your CSS classes to give them more transparency and meaning to other developers.

What it's not

  • An all-singing all-dancing feature rich component library à la Twitter Bootstrap or Foundation.
  • A panacea for all UI builds. Scally doesn't try to be all things to all men, it does one job and one job well, then gets out of your way.
  • Personally I think frameworks have gone too far. Look at Bootstrap, it's not a framework, it's a theme off the shelf. Most frameworks usually do so much more than you actually need, are too bloated and way too prescriptive and opinionated. They can be good for prototyping but most project UI's are unique so you want something that leaves the theming (cosmetics) to you.
  • Scally provides all the building blocks you need and leaves all the theming to you i.e. it doesn't get in your way.
  • We can develop a Campaign Monitor version of Scally that sits above Scally and includes all of CM's styles, this is how we tackled it at Westfield.

Why use it?

The problems with CSS

  • It's old, really old.
  • It operates in a global namespace.
  • It's very, very loose.
  • Source order is critical.
  • Specificity negates the above.
  • CSS started out styling the simplest of HTML documents. It's catching up with some impressive additions however it still lacks a lot.
  • CSS is leaky as fuck! Things can get bloated and very non-DRY very quick.
  • Forced to write CSS differently when it comes to following OOCSS principles, specificity needs to be as flat as possible.

Writing CSS is easy, architecting it isn’t.

Scally helps

  • By providing a solid architecture.
  • By being unopinionated and highly configurable.
  • By being performant (light-weight).
  • By enabling rapid UI development.
  • By ensuring UI's are consistent.
  • You should never have to wonder where something lives as everything has it's place. Meaning your CSS will be highly maintainable.

BEM

The naming convention follows this pattern:

.block {}
.block__element {}
.block--modifier {}
  • .block represents the higher level of an abstraction or component.
  • .block__element represents a descendent of .block that helps form .block as a whole.
  • .block--modifier represents a different state or version of .block.

The driving principle

When I start a new project, I assume it will eventually be open sourced (even if it's unlikely). This mindset leads to effortless modularization. If you think about how other people outside your company might use your code, you become much less likely to bake in proprietary configuration details or tightly coupled interfaces. This, in turn, leads to cleaner, more maintainable code. Even internal code should pretend to be open source code.
